1.  개요

1.1 목적

본 개발 표준 가이드는 다음과 같은 목적을 가진다.

  • 다양한 프로젝트에서 일관된 개발 품질을 유지하기 위한 기준 제공
  • 개발자 간 코드 스타일 통일, 협업 프로세스 정립
  • 유지보수성, 확장성, 보안성을 고려한 지속 가능한 개발 문화 구축
  • 신규 개발자나 외부 인력도 빠르게 적응할 수 있는 공통 기준 문서로서의 역할 수행

1.2 적용 범위

  • 본 문서는 전사적 프로젝트 모든 소프트웨어 개발 활동 전반에 걸쳐 적용된다.

     적용 대상 예시:

  • 웹 애플리케이션 개발
  • 백엔드 서비스 및 API 서버 구축
  • 데이터베이스 설계 및 운영
  • 자동화 도구, 내부 시스템 개발 등

    ※ 단, 실제 사용 기술 스택, 아키텍처 등은 각 프로젝트의 특성과 목표에 따라 추후

정의되며, 본 가이드는 그 기반이 되는 공통 표준을 제공함

3.3 주석 작성 규칙

  • 주석은 무의미한 반복 금지, 핵심 로직 위주로 작성
  • java: javadoc 사용

/**

 * 사용자 정보를 반환합니다.

 * @param userId 사용자 ID

 * @return 사용자 정보 객체

 */

public User getUser(String userId, String userKey) { ... }

  • javaScript / TypeScript: // , /** */ 형식 사용

/**

 * 사용자 이름 반환

 * @param id number

 * @returns string

 */

function getUserName(id: number): string { ... }

3.4 예외 처리 규칙

  • Java: 기본 예외 클래스 ( 예 : IOException, IllegalArgumentException 등) 을 사용
  • throws 키워드로 예외를 명시하거나, 필요 시 try - catch로 직접 처리
  • 파일 업로드, 데이터 변환, 경로 처리 등 실제 실패 가능성이 있는 부분에서 명확히 예외를 던질 것
  • 프론트에서도 API 실패 시 try-catch 또는 .catch()로 처리

3.5 보안 코딩 규칙

  • 입력값 검증 필수( 백엔드 & 프론트 양쪽 모두 )
  • API 요청 시 인증 / 인가 체크 (JWT, 세션 등 사용 시 명확하게 분리 )
  • 절대 로그에 비밀번호 , 개인정보 등 민감 정보 출력 금지
  • 파일 업로드 / 다운로드 시 확장자 , 경로 검증 필수
  • 비밀번호, 핸드폰 번호는 단방향 암호화해서 저장
  • API 응답에도 민감정보는 최대한 포함하지 말것

4.2 패키지 및 모듈 구조

  • 기능별 레이어 ( Controller / Service / Repository ) 분리 원칙
  • 폴더명 / 패키지명은 역할과 기능이 명확하게

- ( 예: member, board, fileupload, auth 등 도메인 중심 )

4.3 공통 모듈 관리

  • 여러 프로젝트에서 재사용할 공통 유틸, 상수, 예외 처리 등은 별도 폴더로 관리

- ex ) utils , common, constants, exception

  • 공통 코드가 많아지면, 별도의 서브모듈 또는 라이브러리화도 검토

4.4 설정 파일 관리

  • 환경별 설정 파일을 구분해서 관리

- Spring: application-local.yml , application-dev.yml, application-ops.yml

- React : .env , .env.development, .env.production

  • 민감정보 ( 키 / 패스워드 등 )는 환경변수 , 별도 보안 파일로 분리

- Git 등 형상관리에서는 무조건 제외 .gitignore

4.5 리소스 및 정적 파일 관리

  • 정적 리소스 ( 이미지, JS, CSS )는 별도 디렉토리에 정리
  • 운영/테스트/개발 환경별로 정적파일 경로, CDN 등도 구분할 수 있음

5. 버전 관리 및 Branch 전략

5.1 버전 관리 도구

  • GitLab을 표준 버전 관리 시스템으로 사용
  • 모든 소스코드, 설정 파일, 빌드 스크립트 등은 GitLab에서 형상 관리
  • 변경 이력, 코드 리뷰, 배포 이력까지 GitLab 기반으로 기록 / 공유

5.2 브랜치 전략

  • 기본 브랜치 구조 예시 :

브랜치명

용도 설명

main

운영 배포용, 실서비스 코드

develop

개발자 통합 브랜치

feature

기능 개발, 이슈별 작업용

  • 브랜치 생성 / 병합 규칙은 팀 상황에 따라 선택 적용
  • 병합 시, Pull / Merge Request 및 코드 리뷰 필수

5.3 커밋 메시지 규칙

  • 일관성 있는 커밋 메시지 작성 필수
  • 기본 포맷 예시

[타입] 간단 설명 (이슈번호, 선택)

  • 타입 예시: feat( 기능 ), fix( 버그 수정 ), refactor , docs( 문서 수정 ) chore( 기타 )
  • 예시 :

[Feat] 회원가입 기능 추가 (#123)

[Fix] 파일 업로드 오류 수정

[Docs] 개발 환경 설정 가이드 추가

  • 커밋 메시지에 변경 목적, 관련 이슈, 영향도 등 명확히 기록

6. API 개발 표준

6.1 API 설계 원칙

  • RESTful API 설계 원칙을 기본으로,

실무 상황에 맞게 자원과 행위 모두 URL에 사용할 수 있음

  • 모든 API는 / api / 도메인 / 액션 형태의 공통 Prefix 구조를 적용
  • HTTP 메서드로 데이터 조회 / 생성 / 수정 / 삭제 등 주요 행위를 구분함

6.2 URI 네이밍 규칙

  • 최상위 Prefix: /api/
  • 도메인(리소스): member, file, user 등
  • 액션 (행위): login, list, register, upload 등
  • 예시 :
  • / api / member / login
  • / api / member / list
  • / api / file / upload

  • 자원 중심의 명사형 URL을 우선 고려하되, 로그인 등 특수한 행위는 동사도 허용

  • JSON 포맷을 표준으로 사용
  • 응답은 항상 아래와 같은 공통 구조 사용

{

  "code": 200,

  "message": "success",

  "data": { ... }

}

  • 에러 발생 시에도 동일한 구조로 code, message를 활용

6.6 API 문서화

  • 모든 API는 PostMan 등 문서화 툴로 관리
  • 문서화 대상
  • 엔드포인트( 경로, HTTP 메서드 )
  • 요청 파라미터: 쿼리 / 바디 / 헤더 / 경로 변수 등
  • 응답 데이터 구조 , 타입, 예시
  • 반환되는 HTTP 상태 코드 및 의미
  • 에러 발생 시 응답 구조와 코드 / 메시지 의미
  • 문서 최신화:
  • 신규 API 추가, 기존 API 수정 / 폐기 시 즉시 업데이트
  • 문서화가 미비된 API는 배포 / 연동 대상에서 제외 가능

6.7 예외 / 에러 처리

  • 에러 응답은 무조건 공통 Response 구조(JSON)로 변환
  • 성공과 동일하게 code, message, data 포맷
  • 에러 코드 / 메시지 관리는
  • ErrorCode(enum 등), MessageUtil ( 클래스 / 상수 ) 등에서 통일 관리
  • 예외 처리 위치 / 방식
  • Controller 에서는 try-catch 또는 Spring의 @ExceptionHandler,

@ConrollerAdvice( 글로벌 예외 처리 ) 사용

  • 예외 발생 시 HTTP 상태 코드도 상황에 맞게 지정
  • 에러 메시지 원칙
  • 사용자 / 개발자가 빠르게 원인 파악할 수 있게 명확하고 구체적으로 작성
  • 내부 구현 / 민감 정보 노출 금지 (예: DB쿼리, 경로, 비밀번호 등)

{

  "code": 401,

  "message": "로그인 인증에 실패했습니다.",

  "data": null

}

7.2 테스트 코드 작성 규칙

  • 테스트 코드도 가독성, 일관성, 네이밍 지켜 작성
  • 파일 / 클래스명은 *Test.java,  *.spec.tsx 등 규칙 지키기
  • 테스트 함수명은 “테스트대상_상황_기대결과” 형태 추천
  • 예: login_유효하지않은비밀번호_예외발생() , login_InvalidPassword_ThrowsException
  • Arrange-Act-Assert( AAA ) 패턴 권장

( 준비→실행→검증 구조로 구분해서 작성 )

7.3 자동화 및 CI 연계

  • 모든 Pull / Merge Request, 주요 브랜치 머지 전 자동 테스트 필수
  • CI 파이프라인 ( Jenkins, GitLab CI 등 )에 테스트, 자동 실행 포함
  • 테스트 실패시 자동 알림 및 빌드 / 배포 차단
  • 주기적으로 전체 테스트 실행

7.4 테스트 데이터 관리

  • 테스트용 데이터는 운영 / 개발 데이터와 분리 관리
  • DB를 쓰는 경우 테스트 전용 DB 사용 권장
  • 테스트 종료 후 데이터 자동 초기화 / 삭제
  • 데이터 생성은 코드 내 Fixture / Factory / Mock 등 활용

7.5 코드 커버리지 측정

  • 테스트 커버리지 ( 코드가 테스트로 검증된 비율 )도 품질 지표로 관리
  • 도구: Jacoco(Java), Istanbul / nyc(JS / TS), SonarQube 등
  • 팀 / 조직 합의로 “커버리지 목표 (예: 70% 이상)”를 정해 관리
  • 커버리지 미달, 테스트 누락 시 알림 / 차단 정책 적용 가능
  • 신규 기능 추가 / 수정 시 반드시 테스트 코드 / 커버리지 포함

8. 빌드 및 배포 표준

8.1 빌드 도구 및 스크립트 관리

  • 프로젝트 언어 / 환경에 맞는 표준 빌드 도구 사용
  • Java: Gradle, Maven
  • JS / TS (React): npm, yarn 등
  • 공통 빌드 스크립트 / 설정은 버전 관리(Git)에 포함
  • 예: build.gradle, pom.xml, package.json, build.sh 등
  • 팀 합의된 빌드 옵션, 의존성 버전, 설정 값은 공통화 / 문서화

8.2 빌드 환경 구분

  • 환경별 빌드 설정 / 파일을 명확히 구분
  • Local: 개발자 로컬 테스트용
  • Dev: 개발 서버용
  • Prod(Ops): 운영 서버(실서비스)용
  • 설정 파일 예시
  • Spring: application-dev.yml, application-ops.yml
  • React: .env.development, .env.production
  • 환경마다 빌드 결과물, DB, 외부 API 주소 등이 다를 수 있으니 반드시 분리

8.3 CI / CD 파이프라인 적용

  • CI( 지속적 통합 ) / CD ( 지속적 배포 ) 파이프라인을 통한 자동화 필수
  • Jenkins, GitLab CI, GitHub Actions 등 사용
  • 주요 단계 예시:
  • Build: 코드 빌드 ( 컴파일, 패키징 )
  • Test: 자동화 테스트 실행
  • Deploy: 서버 ( 개발 / 운영 등 )로 자동 배포
  • Notify: 알림( Slack, Email 등 ) 전송
  • 파이프라인 스크립트( .gitlab-ci.yml, Jenkins-file 등 )는 항상 최신화 / 공유

8.4 배포 자동화

  • 빌드 / 배포 스크립트 표준화
  • 반복 작업 자동화, 수동 배포 최소화
  • 예: deploy.sh, docker-compose.yml 등
  • 운영 배포는 반드시 CI / CD 자동화 또는 표준 매뉴얼 기반 진행
  • 배포 후 버전 태깅( tag ) 및 배포 로그 기록 필수

8.5 Rollback 및 이중 배포 정책

  • 배포 실패 / 장애 발생 시 빠른 롤백( 이전 버전 복구 ) 정책 수립
  • 예: 이전 버전 Docker 이미지, WAS 등 즉시 복구 가능
  • 이중 배포 전략 도입 검토
  • 무중단 배포, 실사용자 영향 최소화
  • 배포 / 롤백 과정은 자동화 / 문서화 하여 신규 인력도 쉽게 따라할 수 있도록 기록

9. 로그 및 모니터링 규칙

9.1 로그 레벨 정책

  • 로그 레벨 ( Log Level )은 중요도에 따라 구분
  • ERROR: 심각한 오류 ( 장애, 서비스 중단 등 )
  • WARN: 경고 ( 잠재적 문제, 처리 가능 )
  • INFO: 일반 정보 ( 시스템 흐름, 상태 )
  • DEBUG: 상세 정보 ( 개발 / 디버깅용 )
  • TRACE: 매우 상세한 추적 ( 보통 기본 비활성화 )

9.2 로그 포맷 규칙

  • 로그 포맷( 형식 )은 팀 표준에 맞춰 통일
  • 예시: [날짜시간] [레벨] [클래스명.메서드] - 메시지
  • 또는 JSON 형태

                { “time” : "2025-07-09T10:00:00", “level” : “INFO”, “service” :        

                “UserService”, “message” : “로그인 성공”}

  • 필수 항목: 시간, 레벨, 위치(클래스 / 메소드), 메시지,(필요시 사용자/트랜잭션 정보 )

9.3 민감 정보 로그 제한

  • 로그에 비밀번호, 주민번호, 카드번호, 등 개인정보는 절대 기록 금지
  • 예외 메시지 / 오류 로그에 내부 DB 정보, 경로, 시스템 정보도 노출하지 않음
  • 민감 정보가 노출되지 않도록 별도 필터 / 마스킹 로직 적용

9.4 로그 수집 및 저장

  • 로그 파일 / 데이터는 운영 / 개발 / 테스트 환경별로 분리 저장
  • 보존 기간, 로그 용량 등 정책에 따라 주기적 백업 / 삭제 / 압축 적용
  • 서비스 중요도에 따라 중앙 로그 서버 / 시스템 적용 가능

9.5 외부 모니터링 연계

  • 시스템 / 애플리케이션 로그, 성능 지표, 에러는 외부 모니터링 도구와 연동 가능
  • 예:
  • ELK Stack: 로그 수집, 검색 시각화
  • Prometheus / Grafana: 서버 / 서비스 성능, 실시간 모니터링
  • Sentry: 실시간 에러 추적 / 알림
  • 모니터링 대시보드 / 알림 설정으로 장애 조기 탐지

9.6 알림 및 장애 감지

  • 특정 레벨 이상의 로그 (예: ERROR) 발생 시 , Slack/이메일/SMS 등으로 실시간 알림 전송
  • 자동화된 장애 감지 / 대응(서버 재기동, 자동 롤백 등)도 정책적으로 준비
  • 로그 / 알림 정책은 환경, 서비스 중요도에 따라 맞춤

10. 보안 개발 표준

10.1 입력값 검증

  • 모든 외부 입력값은 신뢰하지 않는다
  • 프론트엔드 / 백엔드 모두에서 입력값( 파라미터, 폼 데이터, API Body 등 ) 유효성 검증 필수
  • 예: 빈 값 / 잘못된 형식 / 길이 제한 / 정규식 검사 등
  • 파일 업로드 시 파일명, 확장자 , MIME 타입, 경로 등 철저히 체크일 업로드 시 파일명, 확장자 , MIME 타입, 경로 등 철저히 체크

10.2 인증 및 권한 관리

  • 로그인, 회원관리, 비밀번호 변경 등 민감 기능은 반드시 인증 필요
  • 인증( JWT, 세션 등 )과 인가( 권한체크: 관리자 / 사용자 / 게스트 등 ) 명확히 구분
  • 민감 API, 페이지, 리소스는 권한별 접근제어 반드시 적용
  • 세션 하이재킹, JWT 변조, 토큰 유출 등 대비책 마련

10.3 암호화 및 저장 보안

  • 비밀번호는 무조건 단방향 해시 (bcrypt, SHA-256 등)로 저장
  • 중요 정보( 주민번호 , 계좌, 카드 등 )는 암호화 저장
  • 네트워크 / API 통신은 반드시 HTTPS (SSL / TLS) 사용

항목

10.5 로그 및 예외 정보 보안

  • 에러 로그에 민감 정보( 비번, 주민번호, 내부 경로, 쿼리 등 )절대 노출 금지
  • 사용자에게 반환하는 에러 메시지는 일반화 내부 사유, 시스템 구조 등은 숨김

10.6 외부 라이브러리 및 종속성 관리

  • 오픈소스 / 외부 라이브러리 사용 시 최신 보안 패치 적용
  • 취약점 분석 도구( npm audit, OWASP Dependency-Check 등 )로 주기 점검
  • 공식 저장소, 신뢰도 높은 라이브러리만 사용

10.7 보안 점검 및 테스트

  • 신규 서비스 / 주요 기능 배포 전 보안 점검 필수
  • 코드 정적 분석, 취약점 스캐닝, 모의 해킹 등
  • 배포 후에도 정기적( 분기 / 반기 / 연 1회 등 )으로 보안 점검
  • 결과는 문서화 / 내부 공유, 주요 취약점 발견 시 즉시 보완

11. 문서화 표준

11.1 개발 산출물 관리

  • 프로젝트 진행 중 생성되는 모든 문서는 체계적으로 관리
  • 예시: 요구사항 명세서, 설계서, API 명세서, 테스트 케이스, 회의록 등
  • 문서별 버전 / 날짜 / 작성자 / 변경 이력 명확히 기록
  • 공식 산출물 목록 및 양식은 팀 / 조직 내 공유

11.2 문서 작성 도구 및 저장소

  • 문서 작성 도구 예시
  • Notion, Confluence, Google Docs, 한글, Excel 등
  • 개발용 문서는 MarkDown, Wiki, README 등도 사용
  • 모든 공식 문서는 중앙 저장소 (Git, 문서 서버 등)에 보관
  • 비공식 / 임시 문서도 정해진 폴더나 공유 경로 활용

11.3 API 문서화

  • 모든 API 명세는 PostMan 등으로 자동 / 수동 문서화
  • 엔드포인트, 파라미터, 응답 / 에러 구조, 상태코드, 예시 등 포함
  • 문서화가 안된 API는 배포 / 연동 금지 원칙
  • 문서 URL, 최신 버전 정보는 팀 내 항상 공유 / 공지

11.4 코드 내 주석 문서화

  • 필수 로직, 주요 함수 / 클래스에는 JavaDoc, JSDoc 등 공식 주석 표준 적용
  • 주석은 불필요하게 남발하지 않고, 핵심단어 중심으로 명확하게 작성
  • 반복 코드 / 간단 기능은 주석 생략, 핵심 / 복잡 로직만 친절하게 남김

11.5 README 작성 기준

  • 모든 프로젝트 최상단에 README.md 필수 작성
  • 포함 내용 예시:
  • 프로젝트 개요 / 설명 / 목적
  • 주요 기술 스택
  • 폴더 / 구조 설명
  • 실행 / 빌드 / 배포 방법
  • 개발, 운영 환경 / 명령어 예시
  • 담당자 / 문의처 등
  • 신규 프로젝트 / 외주 / 협력사에도 README 템플릿 공유

11.6 변경 이력 및 버전 관리

  • 중요 문서 / 프로젝트 산출물은 변경 이력 표로 관리
  • 코드 / 문서 모두 Git 등 버전관리 도구 적극 활용
  • ChangeLog, HISTORY.md 문서 상단 이력 테이블 등 활용

13. 기타

13.1 공통 유틸리티 및 상수관리

  •  반복적으로 쓰이는 함수(날짜 변환, 문자열 처리 등)는 공통 유틸리티로 분리해 관리
  • 시스템에서 자주 쓰는 상수( 에러코드, 메시지, 환경 값 등 )는 별도 상수 파일 / 클래스 / enum 으로 일괄 관리
  • 예시: DateUtils, ErrorCode, AppConstants 등

13.2 외부 라이브러리 관리

  • 외부 라이브러리는 팀 / 회사 합의 및 공식 저장소 위주로 선정
  • 오픈소스는 라이센스 검토, 보안 취약점 점검 필수
  • 의존성 버전, 잠금 파일( package-lock.json, yarn.lock 등 )로 팀 전체가 동일 환경에서 개발 / 배포 가능하도록 관리

13.3 사용되지 않은 코드 및 기능 처리 기준

  • 더 이상 사용하지 않은 API / 함수 / 기능은 @Deprecated 어노테이션 또는 명확한 주석 표시 및 주석 처리
  • 미사용 코드, Dead Code, 임시 / 테스트 파일은 정기적으로 정리
  • 관련 내용은 변경 이력 / README / 팀 회의록 등에 명확히 기록

13.4 협업 커뮤니케이션 규칙

  • 협업 도구( Slack, JIRA, Notion 등 ) 적극 활용
  • 이슈 / 업무 / 공유사항은 “담당자 명확화” , “기한 / 진행상태” 등 구체적으로 기록
  • 변경사항 / 공지 / 장애 등은 정기 회의, 공지 채널 등으로 투명하게 공유
  • 문서화 / 회의록 / 리뷰 기록 등 업무 이력은 중앙 저장소에 일괄 관리
